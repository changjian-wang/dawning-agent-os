# AI 合规专题：EU AI Act、NIST AI RMF、中国生成式 AI 管理办法

> 2024-2026 是 AI 监管落地的关键窗口：
> - **欧盟 AI Act** 已立法（2024.08 生效），2026 全面执行。
> - **美国 NIST AI RMF** 成为联邦采购标准。
> - **中国生成式 AI 服务管理办法** 2023 实施，ISO 42001 国际化。
>
> 本文梳理主要法规、企业合规义务、技术落地对照、Dawning Layer 7 的覆盖。

---

## 1. 为什么现在必须看合规

### 1.1 现实压力

- **欧盟市场**：2026 起不合规不能上市
- **政府客户**：NIST AI RMF 是投标门槛
- **中国境内**：生成式 AI 备案要求
- **金融/医疗**：行业监管叠加
- **员工 / 客户信任**：合规 = 品牌资产

### 1.2 不合规代价

- 欧盟：最高 **3500 万欧元或全球营收 7%**
- 中国：备案未通过 → 下架
- 美国：FTC 对虚假 AI 宣传的罚款
- 民事索赔：歧视 / 数据泄漏
- 声誉：负面报道指数级

## 2. 欧盟 AI Act

### 2.1 时间线

```
2021.04  提案
2024.03  欧洲议会通过
2024.08  正式生效
2025.02  禁用 AI 规则生效
2025.08  GPAI 义务生效
2026.08  高风险系统义务生效
2027.08  完全执行
```

### 2.2 风险分级

```
                     ┌──────────────────────────┐
           禁止      │ 社会信用评分、实时生物识别  │  → 禁用
                     │（执法例外）、操纵脆弱群体    │
                     └──────────────────────────┘
                     ┌──────────────────────────┐
         高风险      │ 医疗诊断、招聘、信用评分、  │  → 严格合规
                     │ 执法、教育、关键基础设施     │
                     └──────────────────────────┘
                     ┌──────────────────────────┐
         有限风险    │ Chatbot、深度伪造、情感识别 │  → 透明度义务
                     └──────────────────────────┘
                     ┌──────────────────────────┐
         最小风险    │ 垃圾邮件过滤、推荐系统       │  → 无特殊要求
                     └──────────────────────────┘
```

### 2.3 高风险系统义务

1. **风险管理系统**（持续）
2. **数据治理**（质量、代表性、无偏）
3. **技术文档**（模型卡、数据卡）
4. **日志与 Trace 保留**
5. **透明度与用户告知**
6. **人类监督**
7. **准确性、鲁棒性、网络安全**
8. **合规评估**（CE 标志）
9. **事故报告**

### 2.4 GPAI（通用目的 AI）义务

所有 LLM 提供方：
- 训练数据摘要
- 版权合规声明
- 技术文档

**系统性风险模型**（训练算力 > 10^25 FLOP）额外：
- 模型评估
- 系统性风险评估
- 网络安全
- 事故报告

目前 GPT-4 / Claude Opus / Gemini Ultra 级别属于此类。

### 2.5 透明度要求

- **Chatbot**：必须告知"您在与 AI 对话"
- **Deepfake**：必须标记
- **生成内容**：水印 / 标识

## 3. 美国 NIST AI RMF 1.0

### 3.1 概述

- **非强制法律**，但联邦采购事实标准
- 2023.01 发布，2024.07 发布 Generative AI Profile

### 3.2 四大功能

```
GOVERN（治理）
  ├── 政策 / 责任 / 文化
  └── 供应链、第三方

MAP（识别）
  ├── 上下文 / 风险场景
  └── 利益相关方

MEASURE（测量）
  ├── 定性与定量指标
  └── 红队 / Eval

MANAGE（管理）
  ├── 优先级 / 资源分配
  └── 响应 / 改进
```

### 3.3 生成式 AI Profile 关注

- 数据隐私
- 知识产权
- 虚假信息
- 有害输出
- 环境影响
- 价值链透明

### 3.4 实践映射

| NIST | 企业落地 |
|------|---------|
| GOVERN | AI 委员会、政策、供应商审查 |
| MAP | 场景风险登记、DPIA |
| MEASURE | Golden Set eval、红队、审计 |
| MANAGE | 事故响应、持续监控、改进 |

## 4. 中国生成式 AI 服务管理暂行办法

### 4.1 核心要求（2023.08 施行）

1. **备案制**：面向公众提供生成式 AI 服务需备案
2. **真实身份**：提供者对使用者实名
3. **数据合法**：训练数据合规、知识产权
4. **内容导向**：不得生成违反法律法规内容
5. **安全评估**：上线前评估
6. **生成内容标识**：合成内容需标记
7. **用户权利**：投诉、举报
8. **透明度**：说明服务能力与局限

### 4.2 备案流程

```
立项 ──► 安全自评估 ──► 算法备案 ──► 大模型备案 ──►
公众服务 ──► 持续监管 ──► 年度评估
```

### 4.3 算法备案与大模型备案区别

- **算法备案**（互联网信息服务算法推荐管理规定）：面向推荐 / 合成类算法
- **大模型备案**（生成式 AI 暂行办法）：面向公众生成服务

### 4.4 相关标准

- **TC260 标准**：生成式 AI 服务安全基本要求（2024）
- **信通院认证**：可信 AI

### 4.5 其他相关法规

- **数据安全法**
- **个人信息保护法**（PIPL）
- **网络安全法**

## 5. 国际 ISO / IEEE 标准

### 5.1 ISO/IEC 42001（AI 管理体系）

- 2023.12 发布
- 类似 ISO 27001 但针对 AI
- 建立 AI Management System (AIMS)

**体系覆盖**：
- 风险管理
- 数据治理
- 生命周期管理
- 供应商管理
- 透明度与可解释
- 人员能力

### 5.2 ISO/IEC 23894（AI 风险管理）

- 风险管理指引

### 5.3 ISO/IEC 42005（AI 影响评估）

- Impact Assessment 标准（起草中）

### 5.4 IEEE 7000 系列

- IEEE 7000：伦理设计
- IEEE 7001：透明度
- IEEE 7003：算法偏见

## 7. 隐私法规叠加

### 7.1 GDPR（欧盟）

- AI 系统处理个人数据需合法基础
- **自动化决策**权利（第 22 条）：
  - 用户有权不受单独自动化决策
  - 有权获得人类审查
- **被遗忘权** vs LLM 训练
- DPIA 义务

### 7.2 PIPL（中国）

- 类似 GDPR
- 跨境传输强约束
- 自动化决策透明度

### 7.3 CCPA / CPRA（加州）

- Opt-out 权
- 自动化决策告知

### 7.4 HIPAA（美国医疗）

- PHI 严格保护
- LLM 不得未授权处理 PHI

## 9. 高风险场景示例

### 9.1 招聘 Agent

**风险**：性别 / 年龄 / 种族偏见

**要求**：
- 偏见审计（NYC 144）
- 拒绝理由可解释
- 人类最终决定（EU）
- 候选人知情同意

### 9.2 信贷审批 Agent

**要求**：
- 不可单独自动化决定（GDPR 22）
- 申请人获得决定理由
- 对抗性样本审查
- 公平性指标监控

### 9.3 医疗诊断辅助

**要求**：
- FDA / CE 认证
- 医师最终决定
- 全程 trace 保留
- 错诊责任归属

### 9.4 儿童教育 Agent

**要求**：
- COPPA（美）
- 儿童个人信息保护（中）
- 年龄门控
- 家长可控
- 有害内容严格过滤

## 11. 数据合规核心操作

### 11.1 训练数据

- 合法来源（授权、公开、许可）
- 版权尊重
- 敏感数据排除
- 数据卡文档

### 11.2 推理数据

- 用户数据不被用作训练（opt-out / Enterprise 默认）
- 保留期限明确
- 脱敏后存储
- 加密在传输和静止

### 11.3 跨境

- 欧盟用户数据 → 优先欧盟内处理
- 中国跨境需安全评估
- 使用 Azure/AWS 多区域

## 12. 红队与对抗测试合规

合规要求的红队 ≠ 技术意义的红队。

**合规红队包含**：
- 偏见测试（性别 / 种族 / 年龄）
- 虚假信息
- 有害内容
- 隐私泄漏
- Prompt 注入（安全侧）
- 自主操作失控

**工具**：
- PyRIT
- Garak
- 自研对抗集

## 14. 合规驱动的架构选择

### 14.1 区域隔离

```
EU Client ──► Dawning (EU Region) ──► Azure OpenAI EU
US Client ──► Dawning (US Region) ──► AWS Bedrock US
China Client ──► Dawning (CN) ──► 境内 LLM（通义/文心/DeepSeek）
```

### 14.2 私有化部署

- 高敏感场景 → 完全私有
- Ollama / vLLM 自托管
- 向量库自托管
- 无外呼

### 14.3 混合

- 公开信息 → API LLM
- 敏感信息 → 本地 LLM
- Router 按数据分类自动路由

## 17. 合规不是成本，是护城河

> **合规先行的公司**：
> - 进得了欧盟市场
> - 过得了政府采购
> - 赢得企业客户信任
> - 规避监管罚款
>
> **合规滞后的公司**：
> - 补课成本 10x
> - 错过窗口
> - 承担品牌伤害

Dawning 的立场：**合规不是附加功能，是 Layer 7 内置能力。**
企业不必从零搭合规，Kernel 就自带治理骨架。

## 18. 小结

> 法规不可怕，失控才可怕。
> 把 **EU AI Act / NIST AI RMF / 中国办法 / ISO 42001** 放进架构图，
> 让 Layer 7 的能力对齐法规要求——
> 这是 2026 企业 Agent 不可绕过的路径。
