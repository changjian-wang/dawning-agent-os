# ADR-004 重要性级别与确认机制

> 将 agent 动作分为 L0/L1/L2/L3 四级，明确自决、确认、软删除与永久删除边界。

## 背景

[PURPOSE.md](../../PURPOSE.md) §4.1 已把「重要动作必须显式确认；代笔必须明示」写成产品红线，但 v1.7 仍未定义什么算重要、什么可以自决、什么必须确认。

用户已确认三条方向：

- 查询、搜集、分类、整理等低风险操作，正常情况下可以由 agent 自决。
- 修改文件内容、删除等高危操作需要人确认；删除尤其不能直接硬删，必须提供恢复可能。
- 主动性采用保守默认：只有高置信、低风险、可撤销、能明显省事时才主动执行或提示。

如果没有分级机制，agent 会在两个坏结果之间摇摆：要么事事请示，退化成工具；要么过度自决，伤害信任。

## 备选方案

- 方案 A：所有动作都先请示。
- 方案 B：按动作风险分级，低风险自决，高风险确认。
- 方案 C：不设固定分级，让 agent 每次临场判断。

## 被否决方案与理由

**方案 A 所有动作都先请示**：

- 安全但低效，会把「管家」退化成「按钮很多的工具」。
- 查询、搜集、分类、整理本来就是省心场景，过度请示会抵消产品差异化。

**方案 C 每次临场判断**：

- 缺少稳定边界，future agent 容易根据模型倾向漂移。
- 用户无法形成预期，不知道哪些事 agent 会自己做、哪些事会先问。

## 决策

采用方案 B：按动作风险分为四级。

| 级别 | 动作类型 | 默认处理 |
|---|---|---|
| L0 信息型 / 只读型 | 查询、搜集、读取、总结、分析、候选方案生成 | agent 可自决；必要时汇报结果 |
| L1 可逆整理型 | 分类、打标、整理、本地归档、可恢复移动、建立索引 | agent 可自决；事后告知并保留回滚路径 |
| L2 内容修改型 | 修改文件内容、更新记录、批量重命名 / 移动、写入本地状态 | 需要 user 明确确认；若当前请求已明确指定对象与动作，可视为确认，但必须提供 diff / 操作记录 / 回滚可能 |
| L3 高风险型 | 删除、永久删除、发邮件、付款、对外发言、git push、日历写入、权限 / 密钥变更 | 必须 user 一键确认；不得从模糊指令中推断授权 |

删除规则单独加严：

- 删除默认走软删除 / 回收站 / 隔离区，不直接硬删。
- 默认保留 30 天恢复窗口；到期后才允许进入清理流程。
- 永久删除需要用户再次确认，或用户提前接受清理策略且清理前仍可取消。
- 如果某个环境无法提供恢复路径，则该删除视为不可逆 L3 动作，执行前必须给出备份 / 导出 / 恢复替代方案。

主动性默认规则：

- 只有当动作高置信、低风险、可撤销、能明显省事时，agent 才主动执行或主动提示。
- 其余情况优先给 2–4 个候选方案，让 user 选择 / 否决 / 微调。
- 用户在当前任务中给出的明确授权只覆盖对应对象与动作，不得自动外推到其它文件、批量范围或对外动作。

## 影响

**正向影响**：

- 查询 / 搜集 / 分类 / 整理可以保持管家感，不被确认弹窗拖死。
- 修改内容、删除、对外动作有明确确认边界，降低误操作伤害。
- 信息整理 MVP 有了可执行的安全地基：整理可自决，改内容和删除要受控。

**代价 / 风险**：

- L1 与 L2 的边界需要在 dogfood 中校准，例如批量移动、批量重命名是否仍可恢复。
- 30 天软删除窗口是默认值，不一定适合所有数据类型；后续可按数据敏感度分层。
- 对外写入能力接入后，L3 的确认体验需要单独设计，否则会变得笨重。
