# ADR-005 MVP 主场景选型 = 信息整理

> 第一个端到端可用的 MVP 场景锁定为信息整理，日程与生活决策作为副场景做读侧 / 候选生成。

## 背景

[PURPOSE.md](../../PURPOSE.md) §2 把 dawning-agent-os 定位为「通用个人 AI 管家」，§4.1 定义了七条产品红线，其中差异化最强的两条是：

- 选择题优先于问答题（options-over-elaboration）
- 长期记忆是核心而非可选模块

但 §最小可用形态原本写「日程 / 信息整理 / 生活决策」三选一未决。在没有 MVP 主场景的情况下：

- 任何 ingest 进来的论文 / 框架都难以判断在不在范围内（§3.1 收录范围只能给原则，无法给具体场景边界）。
- 长期记忆模块没有具体应用场景去检验"建模是否够用"。
- 选择题优先这条交互哲学没有真实任务可以验证。
- 后续 ADR-001 / ADR-002 / ADR-003 三个产品哲学根基 ADR 在没有主场景时容易写成空头声明。

本 ADR 决定主场景，并明确副场景的角色边界，让其余开放问题能逐个被产品决策触发。

## 备选方案

三个候选场景（§PURPOSE 原列表）：

- 方案 A：日程安排（读日历 / 写日历 / 提醒）。
- 方案 B：信息整理（本地文件 / 笔记 / 邮件本地副本的归类、打标签、检索）。
- 方案 C：生活决策辅助（吃什么、买什么、看什么、要不要做某事）。

## 被否决方案与理由

**方案 A 日程**：

- 失败不可逆（错过会议、改错时间无法事后补救），与 §4.1 红线「失败时优先保守」冲突，作为 MVP dogfood 风险过高。
- 闭环依赖外部 OAuth + 多家日历服务 API + 时区处理，工程链路长，迭代慢。
- 长期记忆收益慢——agent 用一年可能只学到「周三晚上不接会议」少数硬规则，验证不了记忆建模的差异化。

**方案 C 生活决策**：

- 几乎没有执行闭环（agent 只列 tradeoff、不下结论，§4.1 主客体边界要求），导致 MVP 无法验证「执行层」「判断层」的边界。
- 反馈周期长且主观——用户说"晚饭吃 A"之后是否满意，agent 难以观察，记忆累积慢。
- 差异化感知最强但可信度最弱：用户在最敏感的领域被建议时容易反问「凭什么相信你」，而 MVP 阶段尚无信任基础。

## 决策

**主场景 = 信息整理**：

- 范围：本地文件 / Markdown 笔记 / 邮件本地副本的归类、打标签、检索；agent 学习用户的分类习惯并主动建议。
- 闭环：输入接入 → 意图识别 → 长期记忆读 → 候选生成 → 用户确认 → 动作执行（移动 / 打标 / 写索引）→ 结果回写记忆 → 失败兜底。
- 必须做：每天 dogfood ≥ 3 次；每次操作都喂养 Memory；候选生成必须遵守 §4.1 #4「选择题优先」。

**副场景（侦察兵） = 日程读侧 + 生活决策列 tradeoff**：

- 日程：只读日历，识别冲突，给 2–3 个候选解（推 / 合 / 延），不调用写 API。
- 生活决策：给 2–4 个候选 + 一句话差异，不下结论。
- 共享主场景的 Memory 模块；是否能被复用，决定「通用层抽象」是否成立（与 PURPOSE §通用层抽取联动）。

## 影响

**正向影响**：

- 收录范围从「通用个人助手」收窄到「信息整理为主、日程 / 生活决策辅助」，§3.1 / §3.3 的边界判断有了具体锚点。
- ADR-001 / ADR-002 / ADR-003 三个产品哲学根基 ADR 都能在信息整理场景里找到第一个具体反例 / 正例，避免空头声明。
- Memory 模块得到具体设计场景：episodic（"上周整理过这个目录"）+ semantic（"这类文档归这一类"）+ user model（"我的分类风格偏 X"）。
- 副场景的存在让"哪些代码该抽进 framework"这条 PURPOSE §通用层抽取问题在 MVP 阶段就能开始观察，不必等产品成熟。

**负向影响 / 必须接受的代价**：

- "通用 AI 管家"在 MVP 阶段只能在信息整理一域兑现「通用」，对外宣传需要诚实管理预期。
- 日程 / 生活决策若长期只做读侧，后续展开为完整闭环时可能要重做架构。
- 副场景被执行闭环需求触发升级时，本 ADR 必须 supersede，不能简单扩写。

**对其它 ADR / wiki 结构的影响**：

- 触发起草 hub `pages/hubs/information-curation.md`（待信息整理领域有 ≥ 2 个内容页时建立，避免空 hub）。
- 已落地 ADR-004「重要性级别与确认机制」——信息整理场景几乎全可逆，是最低风险环境去验证动作分级。
