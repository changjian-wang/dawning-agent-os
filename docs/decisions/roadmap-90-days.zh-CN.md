# Phase 1：90 天路线图（构建企业级分布式 Agent Framework）

> 目标：在 90 天内构建一个企业级、分布式、多 Agent 协作，并具备技能持续自演化能力的 .NET Agent Framework。
>
> 范围：以生产可用为前提，覆盖分布式执行、治理、安全、可观测与可量化演进闭环。

## 0. 成功标准（Definition of Done）

到第 90 天，以下每一项必须通过验证。按 10 个维度分组。

### SC-1 分布式架构

- SC-1.1：控制面已部署，管理 Agent 注册、策略存储、技能生命周期和评测触发。
  > 控制面是整个平台的中央大脑。它持有所有已注册 Agent 的目录、约束其行为的策略规则、包含完整版本历史的技能工件仓库，以及触发评测流水线的调度器。没有正常运行的控制面，Agent 无法被发现、技能无法被发布、策略也无法被执行。

  1. **Agent 注册表** —— 集中式目录，每个 Agent 按名称、版本、能力、工具域和健康状态注册。支持 CRUD、版本标记和按命名空间发现。其他平面通过查询注册表确定哪个 Agent 处理哪类任务。
  2. **策略存储** —— 版本化键值存储，保存所有治理规则：工具执行策略（allowlist/denylist/风险门禁）、委托深度限制、记忆 scope ACL、技能晋级门禁。策略按版本不可变，更新创建新版本。生效策略集按环境（dev/staging/prod）按 Run 解析。
  3. **技能生命周期管理器** —— 管理技能工件的完整生命周期：创建、验证（lint + policy）、发布、灰度、晋级、废弃、归档。拥有技能版本图，在每次修订时触发质量门禁，并与 Skill Router 协调激活状态。
  4. **评测触发调度器** —— 调度和分发评测运行：夜间黄金集基准、按需 PR 评测和技能更新后的回归检查。它产生评测请求并收集结果供 SLO 仪表盘使用。

- SC-1.2：运行面以独立进程运行 Agent Worker。Worker 可启停和替换而不丢失数据。
  > Agent Worker 是执行单元。每个 Worker 运行在独立进程（或容器）中，从队列拉取任务，执行后上报结果。核心要求是零停机可替换：你可以滚动发布新版 Worker、排空旧版，正在执行的工作流从最后一个持久化检查点在新 Worker 上恢复继续。

  1. **任务队列消费者** —— 每个 Worker 连接到持久化任务队列（如 RabbitMQ、Azure Service Bus），竞争拉取 AgentTaskEnvelope 消息。消费采用拉取模式，可配置 prefetch 和可见性超时，防止饥饿和重复处理。
  2. **持久化检查点存储** —— 每个执行步骤后，Worker 将检查点（步骤号、中间状态、工具结果）写入持久存储。崩溃或替换时，新 Worker 加载最后一个检查点并恢复，不重新执行已完成步骤。
  3. **租约与心跳协议** —— 每个活跃 Worker 持有有时限的租约并定期向控制面发送心跳。若 Worker 在宽限期内未发送心跳，控制面将其标记为失败并将其在途任务重新分配给其他 Worker。
  4. **优雅排空与滚动部署** —— 当 Worker 收到关闭信号时，停止接受新任务，完成或检查点化进行中的任务，然后干净退出。部署编排器（如 Kubernetes）在保持最低 Worker 数量的同时用新版替换。

- SC-1.3：记忆面提供短期状态（会话级）和长期知识（跨会话）两个独立可扩展的服务。
  > 短期状态处理当前对话的消息历史、工具调用结果和临时上下文。长期知识存储跨会话持久的事实、技能和学到的模式。这两类关注点的访问模式有本质差异（热访问 vs. 温访问），因此必须独立部署和扩展。

  1. **短期状态服务** —— 低延迟、会话级存储（如 Redis、带持久化的内存存储），保存当前对话的消息列表、工具调用结果和临时变量。支持会话结束时基于 TTL 的自动过期，并提供原子读-改-写操作以支持并发步骤执行。
  2. **长期知识服务** —— 支持向量的持久化存储（如 PostgreSQL + pgvector、Qdrant），保存跨会话事实、技能工件和蒸馏知识。通过统一查询 API 暴露语义检索（Embedding 相似度）、元数据过滤和重要性感知排序。
  3. **Scope 隔离层** —— 在 API 边界强制四级命名空间模型（global / team / session / private）。每次记忆读写请求携带 scope token，与 Agent 的 RBAC 配置校验。跨 scope 访问默认拒绝并作为安全事件记录。
  4. **独立扩展接口** —— 短期和长期服务暴露独立的健康端点、连接池和扩展旋钮。扩展一个不需要扩展另一个。部署清单（如 Kubernetes HPA）基于各自指标（延迟、吞吐量、存储）独立对每个服务进行伸缩。

- SC-1.4：三面通过明确异步契约通信（AgentTaskEnvelope、HandoffEnvelope、SkillArtifact、PolicyDecision），禁止直接进程内耦合。
  > 面之间的松耦合在协议层强制执行。每种消息类型有版本化 Schema、唯一关联 ID 和 TTL。这意味着各面可以独立开发、部署和扩展。禁止面之间的直接方法调用 ——所有通信走消息队列或事件总线。

  1. **版本化 Schema 注册表** —— 每种契约类型以语义版本注册。Schema 演进遵循仅追加规则：新字段有默认值，删除字段走废弃周期。消费方对入站消息按预期 Schema 版本校验，不兼容消息用明确错误拒绝。
  2. **Correlation 与 Causation ID** —— 每条消息携带唯一的 `correlationId`（将同一工作流所有消息串联）和 `causationId`（指向触发本消息的消息）。这些 ID 贯穿所有平面，支持从任意单条消息重建端到端追踪。
  3. **TTL 与死信队列策略** —— 每条消息有可配置 TTL。超过 TTL 或处理失败超过最大重试次数的消息路由到死信队列。死信队列被监控，深度超阈值时触发告警。死信消息保留以供人工调查。
  4. **传输抽象层** —— 消息总线（RabbitMQ、Azure Service Bus、测试用内存实现）通过 `IMessageBus` 接口访问。切换传输只需修改 DI 注册，不需任何平面代码变更。集成测试使用内存传输以提速；staging 和 prod 使用持久传输。

- SC-1.5：幂等协议确保重复消息投递不产生重复副作用，通过混沌测试验证。
  > 在任何分布式系统中，消息都可能被投递多次（网络重试、队列重投递）。每个处理器必须使用幂等键来检测和跳过重复。这不是可选项——通过在混沌测试中注入重复消息，并确认工具副作用、状态变更和审计记录没有重复来验证。

  1. **幂等键生成** —— 每条消息在创建时（生产者侧）被分配全局唯一的幂等键。键是（消息类型 + correlationId + 步骤号 + payload 哈希）的确定性哈希。消费方使用此键（而非自己生成的 ID）检测重复。
  2. **去重存储** —— 轻量级、TTL 作用域的键值存储（如 Redis SET + 过期）保存已处理的幂等键。执行任何副作用前，处理器检查存储。键存在则返回缓存响应而不重新执行。键在可配置窗口（如 24 小时）后过期以限制存储增长。
  3. **副作用击剑** —— 对于与外部系统交互的处理器（如发送邮件、写入数据库），幂等检查必须在副作用之前发生，而非之后。若处理器在副作用和键写入之间崩溃，恢复协议重放消息，外部系统自身的幂等机制（或击剑 token）防止重复。
  4. **混沌验证套件** —— 专用测试套件注入：(a) 完全重复消息、(b) 几乎同时的重复消息（竞态条件）、(c) 处理器崩溃重启后的重复消息。套件断言在任何场景下工具副作用、状态变更、审计日志条目和成本记账均未重复。

### SC-2 多 Agent 协作

- SC-2.1：编排器支持顺序、并行、条件分支、重试/补偿四种工作流原语。
  > 这四种原语覆盖了绝大多数真实协作模式。顺序处理流水线。并行处理扇出/扇入。条件分支处理决策树。重试/补偿处理含回滚的失败恢复。编排器必须以声明式方式组合这些原语——不允许为每个工作流手写控制流。

- SC-2.2：移交契约原子转移 ownership、上下文快照、剩余预算和工具域边界。
  > 当 Agent A 移交给 Agent B 时，系统必须保证在任意时刻恰好有一个 Agent 持有任务所有权。移交信封包含冻结的上下文快照（让 B 知道 A 做了什么）、剩余步数和 token 预算（防止 B 超支）、以及工具域边界（B 只能访问被授权的工具）。原子性意味着不存在任务同时被两个 Agent 或无人持有的窗口。

- SC-2.3：委托深度按工作流可配置。超出深度触发策略违规并中止。
  > 没有深度限制，Agent 链可以无限委托下去，导致成本和延迟失控。深度限制是安全机制：按每个工作流定义设置（例如最多 3 次移交），超出时编排器发出 PolicyViolation 事件、停止工作流，并记录完整的委托链以供调试。

- SC-2.4：共享记忆命名空间模型强制执行：global、team、session、private。Agent 不能读取未授权的 scope。
  > 记忆隔离对多 Agent 安全至关重要。global scope 存放组织级知识。team scope 存放项目级上下文。session scope 存放当前会话状态。private scope 存放 Agent 内部临时数据。访问控制在记忆面 API 层强制执行——Agent 请求未授权的 scope 会收到 access-denied 错误，而非空结果。

- SC-2.5：5 Agent 确定性场景可稳定完成，全链路移交可审计（who, what, when, why）。
  > 这是集成验收测试。5 个 Agent 在预定义任务上协作（如审计 -> 修复 -> 测试 -> 评审 -> 报告）。测试验证：所有移交都记录了 Agent 身份、转移的上下文、时间戳和原因；最终输出正确；整个链路可仅从审计日志完整重建。

### SC-3 Stateful Prompt 协议

- SC-3.1：每次运行产出版本化的 StatefulPrompt 记录，包含系统指令、已选技能、记忆上下文和工具定义。
  > StatefulPrompt 是一次 Agent 运行的完整输入规格。它包含 LLM 将看到的一切：定义行为的系统指令、路由器选出的技能工件、从短期和长期存储检索到的记忆上下文、以及本次运行可用的工具定义。版本化意味着每个 Prompt 都被分配单调递增的修订号并存储，以确保可复现。

- SC-3.2：执行开始后 StatefulPrompt 不可变。变更只在两次运行之间通过演化管道发生。
  > 执行期间不可变保证了确定性重放：给定相同 StatefulPrompt 和相同 LLM 响应，Agent 产出相同结果。这对调试、评测和合规至关重要。任何 Prompt 改进（新技能、更新的指令）只在下一次运行中通过技能演化 Write 阶段生效。

- SC-3.3：StatefulPrompt Schema 向后兼容，历史 Prompt 可在新版运行时重放。
  > 随着框架演进，StatefulPrompt Schema 可能新增字段。向后兼容意味着：(1) 缺少新字段的旧 Prompt 使用安全默认值；(2) 新版运行时可以无错误地反序列化和执行旧 Prompt。通过维护历史 Prompt 语料库并在每次构建时重放来测试。

- SC-3.4：同一 Agent 连续两次运行的 StatefulPrompt diff 可用于变更归因。
  > 当 Agent 行为在两次运行间发生变化时，diff 精确展示哪里变了：是新增了技能？是注入了新的记忆事实？还是修改了指令？这是诊断回归和理解技能演化影响的主要工具。diff 是结构化的（字段级别），不是原始文本 diff。

### SC-4 技能路由器（Read 阶段）

- SC-4.1：路由器接受当前 Prompt 状态、任务描述和可选用户提示，返回 top-k 评分技能。
  > 路由器是 Memento-Skills 学习闭环的 Read 阶段。输入是当前请求的完整上下文：StatefulPrompt（含 Agent 指令和记忆）、用户的任务描述、以及显式提示（如“用 SQL 技能”）。输出是排序后的 top-k 最相关技能列表，每个附带数值评分。这替代了将所有技能注入 Prompt 的朴素做法，后者浪费 token 且干扰 LLM。

- SC-4.2：评分特征包含：语义相似度、历史成功率、上次成功时间、失败模式匹配、声明式 when-to-use 元数据。
  > 路由器使用多信号评分模型。语义相似度通过 Embedding 匹配任务描述与技能 intent 和 when-to-use 字段。历史成功率反映该技能导致成功结果的频率。时间近因性偏向最近成功的技能。失败模式匹配对已知失败模式与当前上下文匹配的技能降权。声明式 when-to-use 元数据提供技能作者的显式指导。这些信号以可配置权重组合。

- SC-4.3：置信度阈值可配置。低于阈值时回退到全量技能注入或向用户请求澄清。
  > 并非每次路由决策都有高置信度。当最高评分技能低于阈值时，系统有两种回退策略：(1) 注入所有可用技能让 LLM 自行选择（安全但消耗 token）；(2) 向用户请求意图澄清（交互场景更佳）。阈值和回退策略按 Agent 可配置。

- SC-4.4：路由器支持在线反馈：每次运行后被选技能收到效用信号（成功/失败/部分、延迟、成本）。
  > 这是路由器在不更新 LLM 权重的前提下实现学习的机制。每次运行后，结果（成功、失败或部分成功）连同运行指标（延迟、token 成本）作为效用信号反馈给被选中的技能。随时间推移，该信号通过更新历史成功率和失败模式特征来改善未来路由决策。

- SC-4.5：路由 top-5 命中率在内部路由基准（>= 30 多样化场景）上 >= 85%。
  > 这是定量验收门禁。路由基准由至少 30 个跨不同领域（代码、数据、搜索、通信等）的任务组成，每个任务有已知 ground-truth 技能。路由器的 top-5 结果必须至少 85% 的时间包含正确技能。该基准在每次构建时运行以检测路由退化。

### SC-5 反思式技能演化（Write 阶段）

- SC-5.1：执行后，反思流水线生成轨迹摘要：执行步骤、使用的工具、成功点、失败点及根因。
  > 反思流水线在每次 Agent 运行后（或失败时）触发。它使用 LLM 分析执行轨迹，产出结构化摘要：执行了哪些步骤、调用了哪些工具（含参数和结果）、哪些步骤成功、哪些失败、以及失败的根因假设。该摘要是技能改进的原始素材。

- SC-5.2：若反思识别到可改进技能，生成候选技能补丁（结构化 Markdown diff）。
  > 并非每次反思都导致技能更新。流水线评估失败或次优结果是否归因于技能本身（而非 LLM 错误或外部服务故障）。若是，生成候选补丁：针对当前技能 Markdown 的结构化 diff，可能更新 instructions、examples、failure-patterns 或 limitations 字段。补丁此时仅生成——尚未应用。

- SC-5.3：候选补丁经过质量门禁后方可晋级：lint 检查、策略合规、黄金集回归评测、可选人工审批。
  > 这是防止坏技能更新进入生产的治理层。Gate 1（lint）：更新后的技能 Markdown 必须正确解析且包含所有必填字段。Gate 2（策略）：技能内容不得违反安全策略（如不允许无限制 shell 访问）。Gate 3（回归）：更新后的技能不得在黄金评测集上使性能退化。Gate 4（人工审批）：可选地，由人类审查者批准变更。任一门禁失败即拒绝补丁。

- SC-5.4：通过的补丁作为新技能修订发布，附完整修订元数据（author: system/human, trigger: failure/optimization, parent revision, timestamp）。
  > 所有门禁通过后，补丁成为新的不可变技能修订。修订元数据记录：作者是系统（自动反思）还是人类；触发原因是特定失败类型还是主动优化；衍生自哪个父修订；以及精确时间戳。这些元数据对审计和回滚至关重要。

- SC-5.5：被拒绝的补丁记录拒绝原因并关联原始运行记录，便于事后分析。
  > 被拒绝的补丁不会被静默丢弃。每次拒绝都记录：候选 diff、由哪个门禁拒绝、具体原因（如“任务 #17 回归：成功率从 90% 降至 75%”），以及原始运行的链接。这使工程师可以审查拒绝模式、识别系统性问题、并手动应用自动流水线无法完成的改进。

### SC-6 技能生命周期管理

- SC-6.1：每个技能是结构化 Markdown 工件，必填字段：intent、when-to-use、limitations、failure-patterns、examples、revision-metadata。
  > 技能工件是框架中的知识单元。通过标准化结构化 Markdown，技能对人类可读、对 LLM 可解析、对版本控制可管理。必填字段确保每个技能有明确目的（intent）、使用指导（when-to-use）、已知边界（limitations）、历史失败模式（failure-patterns）、具体使用示例（examples）和完整变更历史（revision-metadata）。缺少任何必填字段的技能无法通过验证，不能注册。

- SC-6.2：技能存储在版本化注册表中，任意历史版本可检索和比对。
  > 技能注册表是仅追加的：每次更新创建新修订，永不覆盖。这意味着你可以始终回答“这个技能 3 月 15 日是什么样子？”和“修订 7 和修订 8 之间变了什么？”。历史检索支持调试（“这个任务上周还能跑现在失败了——什么变了？”）和合规（“给我看这个事故发生时生效的技能”）。

- SC-6.3：支持灰度发布：新技能修订按可配置比例承接流量，再决定全量晋级。
  > 灰度发布是技能更新的可控上线机制。新修订发布后，初始只处理小比例的匹配请求（如 5%）。若灰度指标（成功率、延迟、成本）在可配置观察窗口内保持健康，比例逐步提升直至全量。这防止一个坏技能更新同时影响所有用户。

- SC-6.4：灰度指标（成功率、延迟、成本）劣化超阈值时自动触发回滚。
  > 回滚是灰度发布的安全网。若灰度修订的成功率低于阈值（如比前一修订差 10%），或延迟/成本飙升超出限制，系统自动回滚到上一个稳定修订。回滚是即时的（≤ 1 分钟），并记录触发回滚的劣化指标，同时向平台团队发告警。

- SC-6.5：技能删除或废弃遵循治理流程：标记废弃 -> 宽限期 -> 归档。无审计记录不允许硬删除。
  > 技能积累了使用历史且可能被运行中的工作流引用。硬删除会破坏可复现性和可审计性。废弃遵循三阶段流程：(1) 标记废弃——技能仍可用但会发出警告；(2) 宽限期——可配置时长（如 30 天），供消费方迁移；(3) 归档——技能转入冷存储，不再可路由，但仍可用于审计和重放。

### SC-7 LLM Provider 层

- SC-7.1：至少实现 3 个 Provider：1 个本地（如 Ollama）、1 个公共云（如 OpenAI）、1 个企业云（如 Azure OpenAI）。
  > Provider 多样性对成本优化、延迟优化、数据驻留合规和故障容错至关重要。本地 Provider（Ollama）支持离线开发和敏感数据场景。公共云 Provider（OpenAI）提供最新模型。企业云 Provider（Azure OpenAI）满足企业合规和 SLA 要求。三者必须通过同一套 Provider 契约测试。

- SC-7.2：统一流式事件模型覆盖：TextDelta、ToolCallRequested、ToolCallCompleted、RunCompleted、Error。
  > 不同 LLM API 的流式格式各异。统一事件模型将它们归一化为 Agent 循环消费的五种事件。TextDelta 传递增量文本。ToolCallRequested 信号 LLM 想调用工具（含名称和参数）。ToolCallCompleted 携带工具结果。RunCompleted 信号最终答案。Error 携带 Provider 级错误。这种归一化使 Agent 循环完全与 Provider 无关。

- SC-7.3：Provider 故障自动降级：主 Provider 出错或超时后在同一 Run 内自动切换到备用。
  > 降级在 Agent 执行循环内透明发生。若主 Provider 返回 HTTP 错误（429、500、503）或响应延迟超出配置的 SLO（如 10 秒），请求自动转发到降级链中的下一个 Provider。重试使用相同 Prompt 和工具。降级事件记录在运行追踪中，包含原始错误和使用的备用 Provider。

- SC-7.4：每次 Provider 调用采集 token 用量、延迟和估算成本，并按 Run 聚合。
  > 每次 LLM API 调用产生 TokenUsage 记录：prompt token 数、completion token 数、总 token 数、模型名和美元成本估算。延迟从请求开始到最后一个字节接收计时。这些按调用存储并按 Run 聚合，支撑“每 Agent 每天成本”和“按 Provider 的 p95 延迟”等仪表盘。成本估算使用按模型可配置的价格表。

- SC-7.5：新增 Provider 只需实现 1 个接口 + 注册 1 个 DI 扩展，不修改 Core 代码。
  > Provider 可扩展性是一等设计目标。新 Provider 作者实现 `ILLMProvider`（含 `ChatAsync`、`ChatStreamAsync`、`ChatStreamEventsAsync` 方法），编写一个 `AddXxxProvider()` DI 扩展，框架在启动时自动发现。无需修改 Agent 循环、编排器或任何其他 Core 组件。通过 Provider 契约测试套件验证。

### SC-8 记忆系统

- SC-8.1：短期记忆支持 buffer、滑动窗口、摘要压缩三种策略，按 Agent 可配置。
  > 短期记忆管理当前对话的消息历史。Buffer 模式保留所有消息（简单但 token 昂贵）。滑动窗口只保留最近 N 条（有界但可能丢失早期上下文）。摘要压缩通过 LLM 将较旧消息浓缩为摘要（保留关键事实同时控制 token）。策略按 Agent 通过配置选择，因为不同 Agent 的上下文需求不同。

- SC-8.2：长期记忆支持语义向量检索，Embedding Provider 可配置。
  > 长期记忆存储跨会话持久的事实、技能和模式。检索通过语义相似度完成：将查询 Embedding 化，返回存储中最近邻向量。Embedding Provider（如 OpenAI text-embedding-3-small、Ollama nomic-embed-text）可配置且可替换，无需修改记忆存储实现。

- SC-8.3：检索评分结合语义相似度、时间衰减（指数衰减）、重要性权重，三者权重可配置。
  > 纯语义相似度不足以实现高质量召回。语义相关但存储于 6 个月前的事实可能已过时。用户标记为“关键”的事实应排名更高。复合评分公式为：`score = w_semantic * similarity + w_recency * exp(-decay * age_days) + w_importance * importance_level`。三个权重和衰减常量均按 Agent 可配置。

- SC-8.4：跨会话检索验证：会话 A 存入的事实在会话 B 中同一 Agent 可检索，测试集召回率 >= 90%。
  > 这是长期记忆的定量验收测试。维护至少 50 个事实-查询对的测试集。第一阶段存入事实；第二阶段（不同会话）发起查询。事实必须至少 90% 的时间出现在 top-5 结果中。该测试在每次构建时运行以捕捉召回退化。

- SC-8.5：上下文窗口使用量超阈值时自动触发压缩，压缩保留关键事实需通过 diff 测试验证。
  > 随着对话增长，上下文窗口逐渐填满。当使用量超过阈值（如模型限制的 80%）时自动触发压缩：较旧消息被摘要，冗余事实被去重，低重要性条目被裁剪。diff 测试验证关键事实（标记为重要或被高频召回的）在压缩后仍然存在，防止无声知识丢失。

### SC-9 安全、合规与治理

- SC-9.1：RBAC 强制执行覆盖：工具执行、技能发布、技能审批、Agent 部署、记忆 scope 访问。
  > 基于角色的访问控制在企业 Agent 框架中不是可选项。五个域被保护：(1) 工具执行——只有拥有正确角色的 Agent 才能调用给定工具；(2) 技能发布——只有授权账户能提交新技能修订；(3) 技能审批——只有指定审查者能批准技能晋级；(4) Agent 部署——只有平台运维能部署或扩缩 Agent；(5) 记忆 scope 访问——Agent 只能访问其角色被分配的记忆 scope。

- SC-9.2：每次运行、移交、技能修订和策略决策记入不可篡改追加式审计日志。
  > 审计日志是合规和事件调查的权威记录。它捕获：每次 Agent 运行（输入、输出、步骤、成本）、每次移交（源 Agent、目标 Agent、转移的上下文）、每次技能修订（diff、作者、通过的门禁）、每次策略决策（允许/拒绝、匹配的规则）。日志是仅追加且防篡改的（如哈希链接条目），支持时间范围查询和全文检索。

- SC-9.3：日志与追踪中敏感数据（PII、密钥）自动脱敏，脱敏规则可配置。
  > Agent 运行可能处理敏感数据：邮箱地址、API 密钥、个人姓名、财务数据。在任何数据到达日志或追踪存储之前，脱敏管道应用可配置规则（正则模式、命名实体检测）。被脱敏的字段用 `[PII:email]` 等 token 替换，保留结构但不暴露内容。脱敏规则可通过配置更新而无需重新部署。

- SC-9.4：工具执行策略支持 allowlist、denylist 和基于风险等级的确认门禁。高风险工具需显式人工审批。
  > 工具风险等级不同。只读搜索工具是低风险的。发送邮件或修改数据库的工具是高风险的。工具执行策略框架支持三层：(1) allowlist——只有明确允许的工具可以运行；(2) denylist——特定工具无论其他规则如何均被阻断；(3) 风险等级门禁——标记为 High 的工具在执行前需要显式的人工审批步骤。

- SC-9.5：技能演化 Write 阶段受策略防火墙约束。包含不安全模式（如无限制网络调用、沙箱外文件写入）的技能被阻断。
  > 技能演化管道可以自主生成新技能内容。若无策略防火墙，它可能无意中创建泄露数据、写入主机文件系统或发起不受限 API 调用的技能。防火墙扫描每个候选技能补丁的不安全模式（可配置规则集），匹配即阻断。被阻断的技能记录具体违规以供审查。

- SC-9.6：合规证据可导出供审计：谁审批了哪个技能、当时生效的策略、访问了哪些数据。
  > 企业客户需要证明监管合规（SOC 2、ISO 27001、GDPR）。合规导出生成结构化报告，覆盖：技能审批链（提交者、审查者、时间戳、门禁结果）、审批时的活动策略快照、以及给定时间范围的数据访问记录。导出格式支持机器可读（JSON）和人类可读（PDF），供审计员审查。

### SC-10 可观测性、SLO 与发布门禁

- SC-10.1：OpenTelemetry Trace 覆盖每次运行，span 层级：run -> step -> provider call / tool call / memory call。
  > 每次 Agent 运行产出一棵追踪树。根 span 是运行本身（含 run ID、Agent 名称、输入哈希）。子 span 表示每个执行步骤。每个步骤进一步包含：LLM Provider 调用（含模型、token 数、延迟）、工具调用（含工具名、参数、结果、耗时）、记忆调用（含操作类型、scope、命中/未命中）。这种层级支持从“这次运行很慢”下钻到“步骤 3 的这个工具调用花了 8 秒”。

- SC-10.2：指标采集包含：运行成功率、步数、token 用量、成本、延迟（p50/p95/p99）、路由命中率、技能演化收益。
  > 指标是 SLO 执行和容量规划的基础。运行成功率跟踪产出正确最终答案的运行百分比。步数跟踪执行效率。token 用量和成本跟踪资源消耗。延迟百分位（p50/p95/p99）跟踪面向用户的性能。路由命中率跟踪技能选择质量。技能演化收益跟踪技能更新随时间的可量化改进。所有指标按 Agent、工作流和环境打标签。

- SC-10.3：SLO 目标已定义并强制执行：
  - 分布式工作流成功率 >= 90%。
    > 至少 90% 的多 Agent 工作流必须成功完成。工作流被视为成功是当其最终输出与评测框架判定的预期结果匹配。失败包括：Agent 错误、工具失败耗尽重试、策略违规和超时。
  - 多 Agent 移交准确率 >= 95%。
    > 至少 95% 的移交必须正确转移上下文和所有权，不丢失数据或重复。准确率通过将移交信封内容与评测场景中每个移交点的预期上下文比对来衡量。
  - Skill Router top-5 命中率 >= 85%。
    > 在路由基准中，正确技能必须至少 85% 的时间出现在路由器的 top-5 结果中，确保 Agent 持续获得正确技能。
  - 演化收益：每季度基线相对提升 >= 10%。
    > 技能演化必须产生可量化改进。每季度使用当前技能重新运行黄金评测集，与 90 天前的基线技能对比。总体任务成功率的相对提升应 >= 10%。
  - P95 端到端延迟 <= 15s（标准工作流）。
    > 95% 的标准工作流（定义为 <= 5 步、<= 3 次工具调用）必须在 15 秒内完成，包括 LLM 推理、工具执行、记忆检索和编排开销。
  - 可用性 >= 99.9%（压测窗口）。
    > 在 staging 压测窗口（持续 2 倍预期峰值负载）期间，平台必须保持 >= 99.9% 可用性。可用性定义为：在延迟 SLO 内收到有效响应（成功或优雅错误）的请求百分比。
  - 技能更新安全性：进入生产的关键策略违规数为 0。
    > 任何违反关键安全策略的技能修订都不得到达生产环境。这是绝对门禁：即使一个违规也是发布阻断项。违规由策略防火墙规则（SC-9.5）定义。

- SC-10.4：发布晋级（dev -> staging -> prod）在任意 SLO 退化超阈值时自动阻断。
  > 晋级流水线在每个环境边界运行完整评测套件。若目标环境中任何 SLO 指标劣于阈值（如成功率比上一次发布下降超过 2%），晋级自动阻断。阻断原因在 CI/CD 仪表盘中报告，包含具体指标值和被违反的阈值。

- SC-10.5：夜间评测产出报告，按技能、Agent、工作流分维度可下钻，报告可版本化可 diff。
  > 每晚评测框架运行完整黄金集和所有路由基准。产出的报告包含：按技能的成功率和成本；按 Agent 的执行画像（步数、延迟、token）；按工作流的端到端指标。报告附带版本号，可与任意历史报告 diff，展示趋势、回归和改进。

## 1. 产品定位与非目标

### 1.1 产品定位

你要做的是框架，不是单一助手应用。

- 关注可复用抽象与稳定扩展点。
- API 保持极简、默认值合理。
- 优先保障可靠性与可测试性。

### 1.2 v0.1 非目标

- 不做大而全的连接器市场。
- 不在无审批机制下允许生产环境自动改写代码。
- v0.1 不直接上跨地域 active-active（先单地域，v0.2+ 再扩展）。
- 不允许无策略约束的自写技能直接进入生产。

## 2. 技术原则

- 极简 API 面：概念更少，默认更强。
- 纯 DI：所有运行时服务走构造函数注入。
- Core 与 Provider 解耦：Provider 逻辑不泄漏到核心循环。
- 配置驱动：appsettings + 环境变量覆盖。
- 默认安全：MaxSteps、MaxCost、风险确认默认开启。
- 默认可观测：每次 Run 都可追踪、可统计、可审计。
- 天然分布式：异步编排、幂等、可重放作为一等能力。
- 演化受治理：技能自更新必须通过质量与安全门禁。

## 6. 风险与缓解

1. 技能漂移导致可靠性下降
- 缓解：灰度发布 + 修订打分 + 自动回滚。

2. 分布式状态不一致
- 缓解：幂等键、持久化检查点、可重放处理器。

3. Router 过拟合近期任务
- 缓解：离线/在线混合评测 + 多样性约束。

4. 反思环路放大不安全行为
- 缓解：write 前策略防火墙 + 强制安全测试。

5. 多 Agent 并行导致成本爆炸
- 缓解：预算感知编排与自适应并发限制。
